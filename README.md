# Clips Credibility

Research prototype for measuring the credibility of social media clips through a
data-quality model. The repository contains Python scripts and example JSON clips
used to evaluate credibility in a health-related use case about statins and
cholesterol.

The implementation follows the model developed in Sebastian Garcia Parra's
master's thesis, where credibility is treated as a measurable and auditable data
quality construct. The model combines trustworthiness, provenance, clip
equivalence, and Trustworthiness Path Stability to explain why a social media
clip receives a given credibility score.

This repository is experimental. It is intended to document and reproduce the
proof of concept, not to provide a production-ready service.

## What This Prototype Does

The prototype processes normalized social media clips represented as JSON files.
Each clip contains content, metadata, interaction signals, user information,
generated features, and quality metrics.

The current implementation supports:

- normalized clip examples from X, Facebook, Instagram, and TikTok;
- medical-domain feature generation using Gemini;
- quality metric computation for certification, access to source data,
  confidence, trustworthiness, Trustworthiness Path Stability, and credibility;
- provenance consolidation over related clips;
- Google-based search modules for textual and image-based provenance support;
- CSV export of generated features and quality metrics.

```text

    ├── provenance_GoogleCrawler_Image

```

Filename prefixes are used as shorthand in the current dataset:

- `c_`: clips expected to have higher credibility in the use case.
- `nc_`: clips expected to have lower credibility in the use case.
- `x`, `fb`, `in`, `tk`: source platform identifiers.

## Data Model

Each normalized clip is represented as a JSON document with the following main
sections:

- `clip`: social media content, identifiers, dates, geolocation, interactions,
  multimedia data, comments, performance metrics, and extraction metadata.
- `user`: user identifiers, public profile information, biography, contact data,
  and education or experience signals.
- `features.generated_features`: features generated by domain modules.
- `quality_metrics`: metrics computed by the quality model.

The JSON files are updated in place as the pipeline runs. This makes the process
easy to inspect, but it also means that scripts should be run on copies when
reproducibility matters.

## Processing Pipeline

The expected workflow is:

1. Start from normalized clip JSON files.
2. Run feature modules to enrich each clip with medical-domain evidence.
3. Compute quality metrics from those features and platform/user metadata.
4. Reconstruct or consolidate provenance paths when related clips are available.
5. Compute Trustworthiness Path Stability over provenance paths.
6. Compute final credibility.
7. Export features and metrics to CSV for analysis.

The implementation is modular: feature modules and metric modules expose a common
processing function and are called dynamically by orchestration scripts.

## Feature Modules

The feature modules use `gemini-1.5-flash` to extract medical-domain evidence
from the normalized clip and user profile:

- `featureModule_medicalEducation.py`: estimates medical education evidence.
- `featureModule_medicalProfession.py`: estimates whether the profile indicates a
  medical profession.
- `featureModule_medicalVerifiability.py`: estimates whether the user/profile
  includes verifiable medical affiliations, institutions, or links.
- `featureModule_medicalInfluence.py`: estimates influence from comments or
  recommendations.

Generated features are appended to `features.generated_features` with a
confidence value in `[0, 1]`, module metadata, timestamps, and processing time.

## Quality Metrics

The prototype computes the following metrics:

- `access_to_source_data`: direct mapping from the `medicalVerifiability` feature.
- `certification`: weighted combination of education and profession quality.
- `confidence`: weighted combination of influence and normalized audience size.
- `trustworthiness`: weighted combination of access to source data,
  certification, and confidence.
- `trustworthiness_path_stability`: stability of trustworthiness values along a
  provenance path.
- `credibility`: weighted combination of trustworthiness and
  Trustworthiness Path Stability.

The current formulas are intentionally explicit so that the score can be audited.
Weights are part of the proof of concept and should be calibrated for other
domains or larger datasets.

## Provenance And Clip Equivalence

The provenance scripts model how clips may be related across platforms or
publication contexts. The current proof of concept includes:

- explicit provenance consolidation from related JSON clips;
- textual search support through Google Custom Search;
- image-based search support through Google Vision web detection;
- a provenance JSON representation inspired by W3C PROV and PROV-SAID ideas.

In the thesis model, provenance is essential because a clip may gain or lose
credibility as it is copied, transformed, or shared across platforms.

## Setup

Use Python 3.10 or newer.

```bash
python -m venv .venv
source .venv/bin/activate
pip install google-generativeai requests praw tweepy beautifulsoup4
```

Some scripts call external APIs:

- Google Gemini
- Google Custom Search
- Google Vision API
- Reddit API
- X/Twitter API
- Facebook Graph API

Before running the scripts, move credentials to environment variables or another
secure configuration mechanism. Do not commit API keys or access tokens.

Copy `.env.example` to `.env` for local development and load it in your shell, or
export the variables manually:

```bash
cp .env.example .env
set -a; source .env; set +a
```

```bash
export GEMINI_API_KEY="..."
export GOOGLE_API_KEY="..."
export GOOGLE_CSE_ID="..."
export REDDIT_CLIENT_ID="..."
export REDDIT_CLIENT_SECRET="..."
export REDDIT_USER_AGENT="clips-credibility/0.1"
export X_BEARER_TOKEN="..."
export FACEBOOK_ACCESS_TOKEN="..."
```

The current scripts read credentials from these environment variables through
`src/config.py`.

## Security Note

This repository should be treated as a research snapshot. If credentials were
ever committed during prototyping, rotate them before using the repository again.
Future versions should load secrets exclusively from environment variables or a
secret manager.

## Limitations

- The prototype was evaluated on a small set of health-related clips.
- The current domain features are medical-specific.
- Several paths are hard-coded for the thesis proof of concept.
- Scripts update JSON files in place.
- API access to social networks is heterogeneous and may change over time.
- The current clip-equivalence/provenance approach is partial and should be
  extended for multimodal content, screenshots, paraphrases, and translations.

## Related Work

This code supports the master's thesis on credibility measurement in social media
platforms and the QDB workshop paper derived from that work. The thesis contains

# Clips Credibility

Prototipo de investigación para medir la credibilidad de clips de redes sociales
mediante un modelo de calidad de datos. El repositorio contiene scripts Python y
ejemplos de clips JSON usados para evaluar credibilidad en un caso de uso de
salud sobre estatinas y colesterol.

La implementación sigue el modelo desarrollado en la tesis de maestría de
Sebastián García Parra, donde la credibilidad se trata como una construcción de
calidad de datos medible, descomponible y auditable. El modelo combina
trustworthiness, provenance, equivalencia de clips y Trustworthiness Path
Stability para explicar por qué un clip recibe cierto puntaje de credibilidad.

Este repositorio es experimental. Su objetivo es documentar y reproducir la
prueba de concepto, no ofrecer un servicio listo para producción.

## Qué Hace Este Prototipo

El prototipo procesa clips de redes sociales normalizados como archivos JSON.
Cada clip contiene contenido, metadatos, señales de interacción, información del
usuario, features generadas y métricas de calidad.

La implementación actual soporta:

- ejemplos normalizados de X, Facebook, Instagram y TikTok;
- generación de features del dominio médico usando Gemini;
- cálculo de métricas de calidad para certification, access to source data,
  confidence, trustworthiness, Trustworthiness Path Stability y credibility;
- consolidación de provenance sobre clips relacionados;
- módulos de búsqueda en Google para apoyar provenance textual y por imagen;
- exportación CSV de features generadas y métricas de calidad.

## Modelo De Datos

Cada clip normalizado se representa como un documento JSON con estas secciones:

- `clip`: contenido, identificadores, fechas, geolocalización, interacciones,
  multimedia, comentarios, métricas de performance y metadatos de extracción.
- `user`: identificadores, información pública del perfil, biografía, contacto,
  educación y experiencia.
- `features.generated_features`: features generadas por módulos del dominio.
- `quality_metrics`: métricas calculadas por el modelo de calidad.

Los JSON se actualizan en el mismo archivo durante la ejecución. Esto facilita la
inspección, pero conviene ejecutar los scripts sobre copias si se necesita
reproducibilidad.

## Pipeline De Procesamiento

El flujo esperado es:

1. Partir de archivos JSON de clips normalizados.
2. Ejecutar módulos de features para enriquecer cada clip con evidencia del
   dominio médico.
3. Calcular métricas de calidad desde esas features y los metadatos disponibles.
4. Reconstruir o consolidar caminos de provenance cuando existan clips
   relacionados.
5. Calcular Trustworthiness Path Stability sobre esos caminos.
6. Calcular credibility final.
7. Exportar features y métricas a CSV para análisis.

La implementación es modular: los módulos de features y métricas exponen una
función común de procesamiento y son invocados dinámicamente por scripts
orquestadores.

## Módulos De Features

Los módulos de features usan `gemini-1.5-flash` para extraer evidencia médica
desde el clip normalizado y el perfil del usuario:

- `featureModule_medicalEducation.py`: estima evidencia de formación médica.
- `featureModule_medicalProfession.py`: estima si el perfil indica una profesión
  médica.
- `featureModule_medicalVerifiability.py`: estima si el usuario/perfil incluye
  afiliaciones, instituciones o enlaces médicos verificables.
- `featureModule_medicalInfluence.py`: estima influencia a partir de comentarios
  o recomendaciones.

Las features generadas se agregan en `features.generated_features` con un valor
de confianza en `[0, 1]`, metadatos del módulo, timestamps y tiempo de
procesamiento.

## Métricas De Calidad

El prototipo calcula las siguientes métricas:

- `access_to_source_data`: mapeo directo desde `medicalVerifiability`.
- `certification`: combinación ponderada de calidad de education y profession.
- `confidence`: combinación ponderada de influence y tamaño de audiencia
  normalizado.
- `trustworthiness`: combinación ponderada de access to source data,
  certification y confidence.
- `trustworthiness_path_stability`: estabilidad de valores de trustworthiness a
  lo largo de un camino de provenance.
- `credibility`: combinación ponderada de trustworthiness y Trustworthiness Path
  Stability.

Las fórmulas actuales son explícitas para que el puntaje sea auditable. Los pesos
forman parte de la prueba de concepto y deberían calibrarse para otros dominios o
datasets más grandes.

## Provenance Y Equivalencia De Clips

Los scripts de provenance modelan cómo los clips pueden relacionarse entre
plataformas o contextos de publicación. La prueba de concepto actual incluye:

- consolidación explícita de provenance desde JSON relacionados;
- soporte de búsqueda textual con Google Custom Search;
- soporte de búsqueda por imagen con Google Vision;
- representación JSON de provenance inspirada en W3C PROV y PROV-SAID.

En el modelo de la tesis, provenance es esencial porque un clip puede ganar o
perder credibilidad al ser copiado, transformado o compartido entre plataformas.
